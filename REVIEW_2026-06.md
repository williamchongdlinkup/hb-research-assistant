`interactiveBibliography/` (`main.py` 1,636 lines, `index.html` 1,571 lines), corpus
`data/HBBiblio_Dec2025_Complete.csv` (1,833 rows).

This is the **Phase A deliverable**: a severity-ranked audit for your sign-off. Nothing has been

**Scope (agreed):** Trust & accuracy · Search & usability · Robustness & tests · plus a corpus

## Verdict

The app is **well-built and already fairly trustworthy**: the Q&A is genuinely grounded
(retrieve → numbered context → strict "answer only from these entries" prompt → citations mapped
back to real entries), the smart-search interpretation is transparent and user-editable, the
audit pipeline is sophisticated, exports are standard, and the frontend is XSS-safe with graceful
fallbacks throughout. It is close to ready for a discerning research audience.

The gap between "good" and "trustworthy enough to put in front of top-tier researchers" comes
down to a **handful of concrete, mostly small fixes** — led by one real correctness bug: the
flagship "Ask the Literature" feature silently ignores Chinese/diacritic search terms on a
bilingual corpus. The good news is the correct bilingual code **already exists in the file** and
just isn't used by the chat path.

## First, what is NOT wrong (false alarms ruled out during the audit)

These were flagged by a first-pass scan and **verified to be non-issues** — recording them so we
don't waste Phase B on them:

- **Secrets are not leaked.** `.env` was never committed and is gitignored (checked full git
  history). Keys are local/Railway-env only. No action.
- **BibTeX export is UTF-8** (`main.py:1143`), not unspecified — CJK survives.
- **RTF export preserves CJK**, not mangles it: `_rtf_escape` converts non-ASCII to `\uN?` RTF
  unicode escapes (`main.py:1080-1082`) *before* the `ascii` encode, so Chinese/Pāli render in Word.
- **Author search works** — `authors` *is* a populated FTS column (`main.py:236, 308-316`).
- **No SQL injection** — all user values are bound parameters; FTS tokens are quoted/escaped
  (`main.py:419-426, 464, 638`).
- **Frontend is XSS-safe** — `escHtml` is applied everywhere including the LLM answer
  (`index.html:1142`) and URL schemes are validated to http(s) (`index.html:927`).

## Trust & accuracy

### T1 🔴 Q&A retrieval silently ignores Chinese/diacritic terms
`/api/chat` retrieves context via `_retrieve_for_query` (`main.py:1235`), which extracts keywords
with `re.findall(r'\b[a-zA-Z]{3,}\b', query)` — **ASCII letters only**. On a bilingual corpus, a
question containing 人間佛教, 星雲, or a Pāli term like *karuṇā* retrieves on the *remaining English
words only*, or returns nothing. The grounding the whole feature depends on quietly narrows or
empties. This is the single most important trust issue.
**Fix (S):** the correct extractor already exists — `_extract_terms` (`main.py:1389`, Unicode-aware,
keeps CJK runs ≥2 and accented Latin) feeding `_fts_rows` (`main.py:1414`), used by the audit
feature. Point chat at it: `entries = [_entry_to_dict(r) for r in _fts_rows(req.message, 50)]`,
and retire `_retrieve_for_query` as a redundant, inferior duplicate. Net simplification.

### T2 🔴 Empty retrieval still queries the LLM
When extraction yields no terms (a CJK-only question today, or a stopword-only one),
`_retrieve_for_query` returns `[]` (`main.py:1238`) but `chat` still builds an empty corpus block
and calls Gemini (`main.py:1287-1294`). With nothing to ground on, the model may answer from prior
knowledge — exactly what "answer EXCLUSIVELY from the entries" is meant to prevent.
**Fix (S):** if `not entries`, short-circuit and return the canned
*"The HB corpus does not appear to contain sufficient information…"* answer with
`sources_count: 0`, skipping the LLM call. (T1 also shrinks how often this triggers.)

### T3 🟠 The UI overstates how grounded an answer is
`sources_count` is set to the **retrieved** pool size (up to 50; `main.py:1285`) and the UI prints
*"Based on {sources_count} entries from the HB corpus"* (`index.html:1171`). An answer that
actually cited 3 works claims to be "Based on 50 entries." That erodes exactly the trust the
citations are meant to build.
**Fix (S):** report and display the **cited** count (`len(citations)`), or relabel honestly, e.g.
*"Searched N entries · cited M."* Pairs naturally with T2 (show 0 when nothing was retrieved).

### T4 🟠 Citations are bounds-checked, not relevance-checked
`chat` accepts any `[N]` the model emits as long as `1 ≤ N ≤ len(entries)` (`main.py:1300-1305`).
A model that cites a real-but-irrelevant in-range entry passes silently. Full verification is hard
and can be lossy, so this is a **judgement call**: I recommend we treat T3 (honest counts) + T5
(low temperature) as the high-value, low-risk mitigations, and only add active citation
verification if spot-checks during Phase B show real drift. Flagging it so it's a conscious choice.

### T5 🟡 Generation temperature is left at the model default
`_chat_with_rotation` sets only `system_instruction` (`main.py:122-126`) — no temperature, so
grounded Q&A runs at the model's default (typically high), which increases paraphrase drift and
citation slippage. Smart-search structuring (`main.py:622`) is also unset, where determinism is
purely beneficial (the audit/parse calls already pin 0.0–0.2).
**Fix (S):** set a low temperature (~0.2) for chat and 0.0 for the smart-search JSON call.

### T6 🟡 Chat ranking is broad (OR + BM25 top-50)
`_retrieve_for_query`/`_fts_rows` OR all terms and take BM25 top-50 (`main.py:1241, 1420`). Broad
multi-concept questions can let a single common term dominate the top of the pool and dilute the
most on-point works. Smart Search already has richer must/any handling (`_compile_fts`,
`main.py:654`). **Optional (M):** consider light query structuring or reranking for chat. Lower
priority than T1–T3; revisit after those land and we can eyeball real answers.

## Search & usability

### S1 🟠 Journal, book, and publisher are not searchable
The FTS index covers only `title, abstract, authors` (`main.py:233-239, 305-318`), and *every*
LIKE fallback also covers only title+abstract (`main.py:442, 1432`). Searching a journal name
("Journal of Buddhist Ethics"), a book title, or a publisher returns nothing even when matching
entries exist — a real "the tool is missing things" hit for a research audience.
**Fix (M):** add `book_title`, `journal`, `publisher` to the `entries_fts` schema and its
populating `INSERT…SELECT`; broaden the LIKE fallbacks to match. Re-verify Smart Search / Audit /
author search still compile (they build on the same index). A focused, well-contained change.

### S2 ⚪ Plain search can over-narrow long queries
`_plain_fts_match` ANDs every whitespace token (`main.py:464`), so a long natural-language query in
**non-smart** mode (toggle off) can collapse to zero hits. Smart mode mitigates this for most
users. **Optional (S):** note in UI, or relax to a ranked OR when an AND yields nothing.

### S3 ⚪ Hero copy understates the corpus
"Search **six years** of curated scholarship" (`index.html:87`) — the corpus spans **1990–2025**.
Likely a leftover. **Fix (S):** reword (e.g. "decades of curated scholarship" / "1,800+ works").

### S5 ⚪ Tab accessibility polish
Result cards, checkboxes, and buttons carry sensible `aria-label`s, but the three tool tabs
(`index.html:61-72`) are plain buttons without `role="tab"`/`aria-selected`. **Optional (M):** add
tab ARIA + arrow-key navigation for screen-reader/keyboard users. (`last_updated` is already shown
on the home screen — `index.html:473-475` — so corpus freshness is visible. No action there.)

## Robustness & tests

### R1 🟡 No automated tests
Nothing guards FTS compilation, citation parsing, export formatting, or the CJK/empty paths above
— regressions ship blind, and the T1–T3 fixes especially deserve locking in.
**Fix (M):** a small `pytest` suite via FastAPI `TestClient` against a tiny fixture CSV (see Tests
section in the plan): FTS + LIKE fallback, CJK & empty-query retrieval (locks T1/T2), citation
parsing/bounds (mock Gemini), venue search (locks S1), export smoke tests incl. a CJK row, and
`/api/search` param bounds.

### R2 🟡 Turn limit is client-only
`MAX_TURNS = 5` is enforced in the browser (`index.html:1065`) but `/api/chat` (`main.py:1279`)
never checks history length — trivially bypassed, so per-session LLM cost is unbounded.
**Fix (S):** reject or truncate when `len(history)` exceeds the cap server-side.

### R3 🟡 Model name unvalidated at startup
`GEMINI_MODEL = "gemini-3.1-flash-lite"` (`main.py:31`) is never checked; if the ID is wrong or
unavailable on deploy, *every* AI feature fails at request time as a generic 502 with no clear
cause. **Fix (S–M):** a lightweight startup ping (or `/api/health` that does a tiny generate) so
misconfiguration surfaces immediately on deploy, alongside the existing `ai_keys` count.

### R4 🟡 Unbounded LLM input
`/api/chat` (`message` + full `history`) and `/api/audit` (the entire pasted bibliography is sent
to the parse call, `main.py:1561`) impose no length limits. A large paste or crafted history
inflates token cost/latency and is an abuse vector. **Fix (S):** cap message, total history, and
bibliography length (with a clear message when exceeded).

### R5 ⚪ Single shared SQLite connection
`_build_db` opens one `:memory:` connection with `check_same_thread=False` (`main.py:191`) shared
across FastAPI's threadpool workers. The DB is read-only after startup and SQLite's default
serialized mode generally tolerates this, but under concurrency it's a latent risk.
**Optional (M):** a tiny connection pool or per-thread connection if we ever see contention.

### R6 ⚪ Generic error messages
Failures collapse to "AI generation failed. Please try again." (`main.py:1298`), which can't
distinguish a transient quota blip (retry helps) from a config error (retry won't).
**Optional (S):** differentiate at least transient vs. persistent for the user.

## Data freshness (corpus sync)

### D1 🟡 Corpus is a December 2025 snapshot
`CSV_PATH` points at `HBBiblio_Dec2025_Complete.csv` (1,833 rows) and `CORPUS_LAST_UPDATED =
"December 2025"` (`main.py:24, 30`). The master in the project root has moved on (e.g.
`HBBiblio_Dec2025_Restored.csv` with backups dated 2026-06-15). The bundled and master CSVs share
an identical 31-column schema, so this is **copy + validate, not transform**:
1. **You confirm the authoritative master** — `HBBiblio_Dec2025_Restored.csv` vs
   `HBBiblio_CurrentSheet.csv` (curatorial call).
2. Timestamped backup of the current bundled CSV (your release-backup practice).
3. Copy into `data/`, update `CSV_PATH`/`CORPUS_LAST_UPDATED` if the filename/date changes.
4. Validate: row count, UTF-8 (CJK/Pāli intact), year range, `serial_no` uniqueness; boot and
   confirm `/api/stats` + spot searches.

### D2 ⚪ No ingest dedup / year sanity
`_build_db` parses years defensively (`main.py:244-251`) but does not check `serial_no` uniqueness
or flag absurd years. **Fix (S):** add a validation pass (warn on dupes/out-of-range) during the
sync.

shows 1,940 entries / June 2026, and citations now render full venue + page ranges.
